### 1.2 핵심 스크립트 설계

#### ExperimentManager.cs — 실험 전체 흐름

```
상태 머신 (6 states):
  Idle → Relocalization → Setup → Running → Survey → Complete
```

- 싱글톤 패턴 (DontDestroyOnLoad)
- `ParticipantSession`에서 참가자 ID, 조건(GlassOnly/Hybrid), 미션 세트(Set1/Set2) 로드
- 각 상태 전환 시 `EventLogger`에 제어 이벤트 기록
- `OnStateChanged` 이벤트 → ExperimentFlowUI, ExperimentHUD, ExperimenterHUD 구독
- 실험자 원격 제어 UI (다음 단계 진행, 긴급 중지)
- Running 상태 진입 시 `ConditionController.SetCondition()` 호출

#### MissionManager.cs — 미션 진행 관리

```
미션 생명주기 (8 states):
  Idle → Briefing → Navigation → Arrival → Verification →
  ConfidenceRating → DifficultyRating → Scored
```

- 경로 데이터에 연결된 미션 목록 순차 실행 (세트당 5개: A1→B1→A2→B2→C1)
- 미션 타입(A/B/C)별 분기 처리
- 검증 질문 정답 판정 및 로깅
- Verification 후 ConfidenceRating → DifficultyRating 순차 수집
- Scored 후 2초 대기 → 다음 미션 또는 실험 상태 전진
- `OnMissionStarted` (Action\<MissionData\>), `OnMissionCompleted` (Action\<MissionData, bool\>) 이벤트

#### ConditionController.cs — 기기 조건 제어

| 조건 | Glass 출력 | Beam Pro 출력 | 핸드트래킹 |
|------|-----------|--------------|-----------|
| GlassOnly | AR 화살표 + BeamPro WorldSpace | WorldSpace 전환 (글래스 표시) | **ActivateHandRays()** |
| Hybrid | AR 화살표 | ScreenSpace (폰 화면) | **DeactivateHandRays()** |

- 조건 전환 시 글래스/Beam Pro 디스플레이 모드 자동 설정
- `OnConditionChanged` 이벤트 → BeamProHubController, BeamProCanvasController, HandTrackingManager
- `CONDITION_CHANGE` 이벤트 로깅

#### SpatialAnchorManager.cs — 공간 앵커 관리

**매핑 모드**:
- `CreateAndSaveAnchor()`: 카메라 위치에 앵커 생성 → 품질 관찰 (최대 8초, GOOD 즉시 저장)
- `ObserveAnchorQuality()`: `GetAnchorQuality(trackableId)` 폴링
- `SaveMappingToFile()`: `anchor_mapping.json` 저장

**실험 모드 (재인식)**:
- `LoadAllAnchors()`: 저장된 앵커 GUID로 재인식 시작
- GUID별 0.5초 폴링, 30초 타임아웃 (`AnchorRelocState`: Pending → Tracking / TimedOut / LoadFailed)
- `OnRelocalizationDetailedProgress`: 앵커별 상태 업데이트
- `OnRelocalizationCompleteWithRate(float rate)`: 전체 재인식 완료 + 성공률

**백그라운드 복구**:
- `StartBackgroundReanchoring()`: 실패 앵커에 대해 5초 루프 폴링
- `TryRemap()`: 15초마다 실패 앵커 재매핑 시도
- `OnAnchorLateRecovered(string wpId, Transform)`: WaypointManager가 fallback → anchor 전환

**Fallback 패턴**: `Waypoint.Position = anchorTransform ?? fallbackPosition` — 재인식 실패 시에도 미터 수준 정확도로 내비게이션 지속

#### HandTrackingManager.cs — 핸드트래킹 관리

**지연 초기화**: 1초 대기 후 Hand Ray 오브젝트 검색, 실패 시 3회 재시도 (2초 간격)

**ActivateHandRays()** (GlassOnly 조건):
1. Hand Ray 오브젝트 재검색 (미초기화 시)
2. XREAL InputSource → ControllerAndHands 전환
3. InputAction 강제 Enable (`EnsureActionsEnabled()`)
4. Left/Right Hand Ray 활성화
5. `OnHandTrackingStateChanged.Invoke(true)`

**DeactivateHandRays()** (Hybrid 조건):
1. Hand Ray 비활성화
2. InputSource → Controller 복원
3. `OnHandTrackingStateChanged.Invoke(false)`

**비상 바인딩 수정** (`DetectAndFixEmptyBindings`): ActionBasedController 바인딩 0개 감지 시 XREAL 핸드 바인딩을 런타임에서 생성

## 3. AR 화살표 렌더링

### 3.1 화살표 방향 계산

```
현재 위치(P) → 다음 웨이포인트(W)까지의 방향 벡터:
  direction = normalize(W.position - P.position)
  yaw = atan2(direction.x, direction.z) * Rad2Deg

화살표 회전:
  arrow.rotation = Quaternion.Euler(0, yaw, 0)
```

- 웨이포인트 반경(2m) 진입 시 자동으로 다음 웨이포인트로 전환
- 화살표는 참가자 시야 하단 중앙에 고정 (world-locked이 아닌 view-locked)

### 3.2 트리거 적용

| 트리거 | 화살표 변화 |
|--------|-----------|
| T1 (안내 열화) | 6초 떨림(±5° 랜덤 진동) → 3초 완전 소실(blackout) → 복구 |
| T2 (정보 불일치) | 정상 표시 (환경 표지판과의 불일치는 물리적 배치로 구현) |
| T3 (해상도 부족) | 화살표 끝이 부채꼴(±30°)로 퍼지며 반투명 처리 (미션 종료까지 지속) |
| T4 (안내 부재) | "목적지 근처입니다" 텍스트만 표시, 화살표 제거 (미션 종료까지 지속) |

### 3.3 화살표 시각 디자인

- 크기: 시야 내 높이 약 2cm (거리감 고려)
- 색상: 기본 파란색(#3498db), 트리거 시 주황색(#e67e22) 점멸
- 애니메이션: 부드러운 방향 보간 (Slerp, 0.3초)

### 4.2 Layer 1: InteractiveMapController

- 건물 평면도 이미지 위에 경로 오버레이
- **POI 마커 시스템**: 회의실, 자판기, 화장실, 비상구 등 마커 표시
  - 마커 탭 → `POIDetailPanel` 오버레이 표시
  - `BEAM_POI_VIEWED` 이벤트 로깅
- 현재 웨이포인트 기반 자동 줌 + 수동 확대/축소 제스처
  - `BEAM_MAP_ZOOMED` 이벤트 로깅
- 현재 위치 마커 (웨이포인트 기반 추정)
- 다음 목적지 핀

### 4.3 Layer 2: InfoCardManager

- **자동 표시 (Push)**: 현재 위치/미션 맥락에 따라 관련 정보 카드 자동 팝업
  - `auto_shown: true` 플래그로 자동/수동 구분 로깅
- **수동 열기 (Pull)**: 사용자가 관련 정보 카드를 직접 탭하여 열기
- **카드 유형**:
  - POI Detail Card: 방 상세 (수용인원, 장비, 호실번호)
  - Sign/Poster Card: 멀리 있는 안내판/포스터의 고해상도 텍스트
  - Comparison Card (`ComparisonCardUI`): 미션 C의 비교 대상 속성 비교 표
  - Landmark Card: 주변 랜드마크 설명
- 이벤트: `BEAM_INFO_CARD_OPENED`, `BEAM_INFO_CARD_CLOSED`, `BEAM_COMPARISON_VIEWED`

### 4.4 Layer 3: MissionRefPanel

- 미션 브리핑 재열람 가능
- 미션 타입별 보조 정보:
  - A: 목적지 POI 힌트
  - B: 주변 지도 정보
  - C: 비교 속성 목록
- 이벤트: `BEAM_MISSION_REF_VIEWED`

### 4.5 Lift-to-Wake 감지

- Beam Pro의 가속도계/자이로 활용
- 기기를 들어올리는 동작 감지 → 정보 허브 활성화
- `DeviceStateTracker`에서 `BEAM_SCREEN_ON` 이벤트 발생
- 내려놓거나 5초간 움직임 없으면 → `BEAM_SCREEN_OFF`
- 탭 전환 시 `BEAM_TAB_SWITCH` 이벤트 발생

### 4.6 조건별 정보 허브 동작

| 조건 | Beam Pro 화면 | 핸드트래킹 |
|------|-------------|-----------|
| GlassOnly | BeamPro WorldSpace (글래스에서 핸드레이로 조작) | ActivateHandRays() |
| Hybrid | 대기 모드 → lift-to-wake로 정보 허브 활성화 (터치 조작) | DeactivateHandRays() |

### 6.3 EventLogger 설계

```csharp
public class EventLogger : MonoBehaviour
{
    public static EventLogger Instance { get; private set; }

    // 세션 시작 — 파일 생성 + CSV 헤더 쓰기
    public void StartSession(string participantId, string condition, string missionSet)

    // 스레드 안전 이벤트 기록 (CSVWriter 유틸리티 사용)
    public void LogEvent(string eventType, string waypointId = "",
                         Vector3? headRotation = null, string deviceActive = "",
                         int? confidenceRating = null, int? difficultyRating = null,
                         bool? verificationCorrect = null, string beamContentType = "",
                         string extraData = "{}")

    // 조건/미션 자동 추적
    public void SetCondition(string condition)
    public void SetMissionId(string missionId)

    // 세션 종료
    public void EndSession()
}
```

- `CSVWriter` 유틸리티 클래스로 CSV 직접 쓰기
- `HeadTracker.Instance.CurrentRotation`에서 머리 회전 자동 취득 (headRotation이 null인 경우)
- 싱글톤 + DontDestroyOnLoad (세션 수명)
- 각 이벤트 즉시 Flush — 앱 크래시 시 데이터 손실 최소화

### 7.2 전처리 단계

1. **파일 통합**: `data/raw/` 내 모든 개별 CSV를 분석 스크립트가 자동 병합

4. **무결성 검사**: 이벤트 순서, 필수 필드, 웨이포인트 시퀀스 검증
5. **요약 테이블 생성**: 참가자 x 조건별 주요 지표 (완료시간, 미션 정확도 등)

## 9. 기술적 리스크 및 대응

| 리스크 | 영향 | 대응 |
|--------|------|------|
| 글래스 tracking loss (실제) | 트리거 T1과 구별 불가 | T1 트리거는 의도적 시각 효과로 구현, 실제 tracking loss는 별도 이벤트 로깅 |
| Beam Pro 배터리 소모 | 2조건 연속 수행 중 방전 | 조건 간 전환 시 충전 시간 확보, 배터리 잔량 모니터링 |
| 실내 위치 추정 오차 | 웨이포인트 도달 판정 부정확 | 공간 앵커 + fallback 패턴으로 미터 수준 정확도 보장, 웨이포인트 반경 2m |
| 앵커 재인식 실패 | 내비게이션 불가 | 백그라운드 복구(5초 루프) + fallback 위치 자동 사용 + 부분 재인식 진행 옵션 |
| 미션 난이도 불균형 | 세트 간 비교 타당성 저하 | 파일럿 테스트로 정답률 확인 후 조정 |
| 참가자 50분 피로 | 가능성 낮음 | 2조건으로 축소되어 피로 리스크 대폭 감소, 역균형으로 순서 효과 통제 |
